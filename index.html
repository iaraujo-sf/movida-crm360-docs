Posicionamento

Este documento é uma visão de arte do possível — não uma especificação de implementação. Apresenta como as capacidades Salesforce se encaixam no ecossistema Movida, respondendo às perguntas da arquitetura sem prescrever solução final.

O que este documento é
  • Mapa de capacidades por camada (o que é possível, quem opera, como evolui)
  • Benchmark — como empresas similares integram CRM a event buses
  • Guia de decisão — princípios e trade-offs para orientar workshops
O que este documento NÃO é
  • Especificação de campos, objetos ou schemas
  • Definição de payloads ou contratos de eventos
  • Solução final — isso será construído nos workshops

1. Regra de Ouro e Princípios

Regra de Ouro — Definida pela Arquitetura Movida
"Se o CRM precisa SABER que algo aconteceu → EVENTO"
"Se o CRM precisa PEDIR algo agora → API"
Critério de decisão para toda integração presente e futura.

Tradução prática

PadrãoQuando usarCaracterísticaExemplos genéricos
EVENTO (padrão)Fato de negócio ocorreu. Ninguém espera resposta.Assíncrono, desacoplado, resilienteCrédito aprovado, multa gerada, lead criado, contrato assinado
API (exceção)Usuário humano espera resposta para continuar.Síncrono, com timeout e fallbackGerar proposta, simular preço, gerar link pagamento

Princípios complementares

PrincípioImplicação
Event-Driven by DefaultToda integração nova começa como evento. API só se justifica com humano esperando.
Materialização localDados externos chegam via evento e são armazenados no CRM. Vendedor nunca espera chamada síncrona para ver contexto.
Operational vs AnalyticalBigQuery = analytics. CRM = operação. Insights do BigQuery viram eventos quando relevantes.
CRM como participante, não hubSalesforce consome e publica no bus. Não centraliza integrações.
IA sem lock-inIA Movida (customer-facing) e Agentforce (seller-facing) coexistem via eventos.

Por que evolui sem refatoração?

  • Desacoplamento via eventos — publisher não sabe quem consome
  • Schema aditivo — campos opcionais sem breaking change
  • Padrão repetível — "novo dado 360" = Topic → Cloud Run → PE → Flow → Objeto → Related List

9. Cenários Conceituais (Benchmark)

Exemplos genéricos de como empresas com modelos similares (fleet, rental, assinatura) integram CRM a event buses. Não prescrevem implementação — servem como referência de padrão.

Cenário A — Lead chega de canal digital
  1. Portal captura interesse → microsserviço padroniza dados mínimos
  2. Publica evento no bus ("lead criou interesse")
  3. Integration layer transforma e entrega ao CRM
  4. CRM: dedup → cria lead → distribui → notifica vendedor
  5. Vendedor atende dentro do SLA. Se estoura → escalação automática.

Regra aplicada: Lead "aconteceu" → EVENTO.

Cenário B — IA faz handoff para humano
  1. IA conversa com cliente (pré-qualificação, FAQ)
  2. Decide handoff → publica evento com resumo, intenção, sentimento
  3. CRM: encontra/cria lead → vincula histórico → cria tarefa → notifica vendedor
  4. Vendedor lê contexto completo → responde com continuidade

Regra aplicada: Handoff "aconteceu" → EVENTO com contexto rico.

Cenário C — Vendedor gera proposta (ação síncrona)
  1. Vendedor clica "Gerar Proposta" no CRM
  2. CRM chama API síncrona do serviço de propostas
  3. Retorna PDF + valores → CRM atualiza registro
  4. Vendedor envia ao cliente

Regra aplicada: Vendedor "precisa pedir agora" → API (exceção controlada).

Cenário D — CRM informa ecossistema (outbound)
  1. Vendedor fecha negócio no CRM
  2. CRM publica automaticamente via CDC (zero código)
  3. Cloud Run captura → transforma → publica no Pub/Sub GCP
  4. Consumers (billing, contratos, BigQuery) consomem cada um no seu tempo

Regra aplicada: Fato aconteceu no CRM → EVENTO para fora.

Cenário E — Visão 360 (dados de outros domínios)
  1. Sistemas de domínio publicam eventos (multa, cobrança, ticket)
  2. CRM materializa localmente como registros vinculados ao cliente
  3. Vendedor abre ficha → vê tudo, sem esperar APIs externas

Regra aplicada: Pós-venda "aconteceu" → EVENTO → materializa. NÃO consulta via API.

10. Decisões Arquiteturais (ADRs)

Cada ADR documenta o porquê de uma decisão — princípios de design validados.

ADR-001: Pub/Sub API (gRPC) via Cloud Run

Contexto: Movida opera Pub/Sub GCP. SF precisa consumir/publicar nesse bus.

Decisão: Cloud Run como bridge bidirecional via SF Pub/Sub API (gRPC nativo, porta 7443).

Descartados: MuleSoft (custo + redundância), Event Relay (só AWS), REST Callout (unidirecional).

Trade-off: Dev custom (~3-4 sem), mas usa stack que DevOps Movida já domina.

ADR-002: CDC para Replicação, Platform Events para Integração

CDC = "o que mudou" → BigQuery, analytics (automático, zero código)

Platform Events = "fato de negócio" → ação em sistema externo (payload controlado)

ADR-003: Materialização Local (não API on-demand)

Decisão: Dados externos chegam via evento e são armazenados localmente no CRM.

Descartado: API on-demand (latência na tela, dependência de disponibilidade).

Trade-off: Eventual consistency (30-60s) + storage cresce. Mas vendedor tem 0ms e CRM funciona offline.

ADR-004: Identity Resolution por CPF/CNPJ

Decisão: CPF (PF) e CNPJ (PJ) como chaves universais, resolvidos via Data Cloud.

Premissa: CPF precisa estar presente e correto nos eventos. Se não, unificação falha.

ADR-005: Salesforce como Participante, não Hub

Decisão: SF é subscriber/publisher no bus. Não centraliza integrações.

Consequência: Se trocar CRM amanhã, eventos continuam fluindo. Apenas consumer muda.

ADR-006: IA Movida (customer) + Agentforce (seller)

IA Movida: Conversa com cliente (LangGraph/MCPs) — pré-qualificação, handoff.

Agentforce: Assiste vendedor no CRM — next-best-action, resumos.

Integração: Via evento (ConversationHandoff). Sem acoplamento direto.

ADR-007: Data Cloud — BigQuery (inicial) + Eventos (ongoing)

Carga inicial: BigQuery zero-copy connector (sem mover dados).

Após go-live: Streaming via eventos do bus em tempo real.

Perguntas difíceis

PerguntaResposta
"Por que não MuleSoft?"Movida já opera Cloud Run. MuleSoft = custo + ferramenta sem valor incremental.
"E se trocar o Salesforce?"Eventos fluem no Pub/Sub (bus neutro). Qualquer CRM substitui como consumer.
"Eventos se perdem?"DLQ + retry exponencial + replay + alertas. Cada hop com fallback.
"Data Cloud obrigatório?"Não no go-live. Sales Cloud funciona standalone. Data Cloud é paralelo.
"Consultar BigQuery direto?"Não. BigQuery = analytics (latência/custo). Insights viram eventos.
